## Vue部署
开发完成之后直接运行命令
```
npm run build
```

<!-- more -->


之后Vue就会生成一个dist目录，这个目录下就是用于生产环境的项目文件，把这个上传到Web服务器，然后配置一下服务器的路由(比如如果服务器用的是Nginx的话就配置一下Nginx的配置文件)即可完成部署
Nginx是一个开源的Web服务器,启动之后就能对配置文件中的项目开启服务(就是可用了)

## Springboot部署
主要有两种方式，打包成独立的JAR包或WEB包。这两个都是要修改pom.xml文件，然后生成包。生产环境下直接复制包过去，创建服务文件后启动服务即可。
打包使用的基本上是Maven/Gradle(任选其一)

## axios和HTTPS的理解:SSL终止在Nginx
用户其实是在和Nginx做交互,剩下怎么跑是服务器内部的事
就是浏览器和Nginx是HTTPS交互,而Nginx和Springboot交互就是HTTP了,Nginx负责加密和解密
【所以】前端从来不知道后端真实ip,端口和数量
### 进一步理解:Vue里直接往`localhost:8080`发请求合理吗?
**如果是本地开发环境**,前后端在同一台电脑,Axios直接发`http://localhost:8080`完全合理
**如果是生产环境(公司里)**,是域名 -> Nginx(服务器) -> Springboot(服务器),前端直接写的是`axios.get("/api/user")`,根本不写ip和端口
【如果每个axios都写域名和端口】虽然简单直观,但换服务器要大改代码,且开发/测试/生产环境无法区分;后端服务器地址是弄在环境变量baseURL,代理(在vue.config.js里写个proxy代理),Nginx(真正在用的,是Nginx收到前端请求url,然后和springboot交互)的

## Nginx(读法是Engine-X或者就读字母)是一款高性能的HTTP服务器和反向代理服务器(也可解决跨域问题)
校园网的研究生系统用的就是Nginx
是"站在最前面接请求的门卫" -> 浏览器的请求不是直接打到Java程序,而是浏览器->Nginx->Spring Boot/Tomcat,Nginx放在后端前面是为了扛并发、做转发、保护后端服务
抢课Nginx报错:是Nginx先接到请求,发现后面服务挂了,就直接报错,发`502Bad Gateway,504 Gateway Timeout,503 Service Unavailable` -> 这里Nginx还活着,所以才能返回服务不可用
【基础使用很简单】安装,启动,配一个反向代理到后端端口,例如`location/{ proxy_pass http://localhost:8080; }`,就默认配置即可,把它当成一个"请求转发器"
【和cloudflare不同】,cloudflare算是互联网基础设施,Nginx才是自己部署用的
**【跨域问题是浏览器问题不是后端问题,最终Nginx统一部署时不会出现】**,后端加`@CrossOrigin`安全性弱,虽然本地开发阶段能通过前端代理或CORS解决跨域,但是真实情况下最终部署的时候前端和后端通常通过Nginx统一代理到同一域名下,不会有跨域问题
一些面试回答(假设场景是租了台云服务器,装了Nginx,前端丢进去,后端跑在8080,能访问了):
> **你用 Nginx 做了什么?**主要用 Nginx 做反向代理和静态资源托管，对外统一暴露端口，后端服务跑在内网端口
> **是反向代理还是负载均衡?**是反向代理，没有做负载均衡，后端是单实例，负载均衡是多实例场景下才需要的
> **HTTPS 怎么配的?**这个项目主要是学习和功能验证，没有申请域名和配置HTTPS，实际生产环境一般会通过Nginx + 证书来做 HTTPS 终止
> **前后端是怎么转发的?**前端请求统一走 /api 前缀，由 Nginx 将 /api 请求转发到后端服务端口;如果要具体一点,就是Nginx里写一个`location /api/ {proxy_pass http://127.0.0.1:8080/;}`
> **Nginx和前端代理有什么区别?**前端代理主要用于开发阶段解决跨域问题，Nginx 是部署阶段的统一入口，负责请求转发、静态资源服务以及 HTTPS 处理
> **【Vue大坑】Vue是SPA(单页应用)，那刷新页面会不会404?**Vue是单页应用，部署时需要在Nginx中配置路由兜底规则，将未命中的路径统一重定向到index.html -> **【白话解释】**就是说你在vue界面点组件模块的时候，顶上的url变了，但是实际上页面还是原来那个没变，这个时候如果刷新，浏览器就会拿着这个变过的url去请求服务器，这时候就找不到，所以才要兜底，不管是哪个url，刷新了就统一重定向到index.html，其实就是服务器不认识前端切换的这些东西  -> **【工程化表述】**SPA的路由是在前端维护的，服务器只负责返回入口页面，刷新时需要由服务器兜底，将请求统一交给前端路由处理

## Kafka消息队列,是解决异步和高并发削峰的
就是原来系统做那些发邮件、做复杂统计那些耗时长的操作的时候,必须得等到这些操作都做完了,主流程才能返回,这样就很慢;现在是把这个操作发给Kafka,然后Kafka弄个队列慢慢去做,主流程发完指令直接就继续走了,不用等直接返回
【Kafka和主应用是并行的】Kafka就是一个"累活缓冲区",独立于主系统并行运行
Kafka是不参与业务逻辑判断的,和主流程无关

## JAR包和WAR包的区别
jar包是类的归档文件，文件格式与平台无关，可以被Java编译器直接使用，就是封装起来的类。jar包比war包便捷，可在开发时用，适合于独立运行的Springboot应用。

war包是JavaWeb程序打的包，一个war包可以理解为是一个web项目，里面是项目的所有东西。它可以直接运行，如对于Tomcat来讲，将war包放置在其\webapps\目录下，然后启动Tomcat，这个包就会自动解压，就相当于发布了，能用了。建立war包比较耗时，所以war包基本用于发布阶段，适合于和其他JavaEE应用共同运行。

## 三范式3NF
1NF原子化，即不可再分 ; 2NF消除部分依赖，如主键有两列，不能只通过其中一列就确定非主键列 ; 3NF消除传递依赖，即消除A->B, B->C, A->C

## Axios就是一个HTTP客户端
Axios是同构的，可以用相同的代码在浏览器和服务器上运行
- HTTPS需要CA申请证书，要收费。
- HTTPS是加密的，可防止访问到假冒网站;而HTTP是明文传输
- HTTPS加密的是内容,IP是没法加密的:类比快递可以封起来,但是地址不能加密

## Vue往Springboot发请求会有跨域问题，导致请求被拦截
跨域问题只能在后端解决，前端无能为力。
最简单的解决办法：
所有的需要跨域的类或者函数上都加上注解`@CrossOrigin`
就把所有Controller类上都加上`@CrossOrigin`就行
> 或者写Springboot的`@Configuration`，不过就相对麻烦了。
