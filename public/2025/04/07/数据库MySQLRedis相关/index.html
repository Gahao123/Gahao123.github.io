数据库MySQLRedis相关

高并发系统全景图(是靠分层削峰的)

用户 -> Cloudflare/CDN（抗流量洪峰）-> Nginx（入口并发 + 限流） -> Spring Boot（业务并发）-> Redis（高并发数据）-> MySQL（最终数据）
【Redis的核心价值】就是抗高并发,但不是取代MySQL,而是搭配使用
MySQL是关系型数据库,数据存磁盘,速度慢并发有限(磁盘IO拖累/锁竞争严重/写操作互斥)但是持久化;Redis属于内存数据库,数据存内存,速度快并发强,是极快的单线程,但持久能力有限,特别适合【读多写少,计数,状态判断】,用于【抢课名额,秒杀库存,登录状态,验证码】
【但是不能只用Redis】Redis内存有限,数据会丢,不适合复杂查询和长期存储;一般是Redis抗并发,MySQL存最终结果,

数据库的增改查都可以，但删一定要慎重

增改查都是有迹可循、删的话就没了
所以增改查都可以让用户来，但是删一般要考虑让管理员来搞

为什么不要用外键ForeignKey？

实际企业开发中，外键是被严格禁止的，当需要多个表之间进行关联的时候，做法是冗余相关字段，而不是建立外键。
外键会引发级联(即外键对应那个表更新了会使引用了这个外键的表也得跟着更新)，尤其是阿里开发手册中提到：【强制】不得使用外键与级联，一切外键概念必须在应用层解决。
外键会大大降低速度(每次都要检查外键约束+级联是强阻塞，加在一起使数据库性能急剧下降)
当数据量大的时候需要进行分库分表(破千万的时候)，外键会导致分库分表难搞

只有单机低并发业务才可以考虑使用外键来保证一致性

MySQL优化查询性能

  • 加索引，但不要加太多的索引，否则会影响写入性能
  • 查询缓存，对于静态查询有用，动态查询效果不大
  • 大的表分区
  • 避免全表扫描，加索引，用WHERE
  • 不要用子查询，用JOIN
  • 将复杂查询拆成多个简单查询

MySQL安全模式

MySQL默认会开启safe-updates状态，导致非主键条件下无法执行update或者delete命令

查询该状态:show variables like 'SQL_SAFE_UPDATES';
关闭安全模式:SET SQL_SAFE_UPDATES = 0;
开启安全模式:SET SQL_SAFE_UPDATES = 1;

MySQL唯一键的大坑

MySQL底层null值是不等于null值的,MySQL认为null之间无法比较,因此往唯一键里插两个null不会报错
涉及唯一键的字段最好有默认值,且这个默认值最好不是正常业务值 ; 或者unique的列一定要有not null
